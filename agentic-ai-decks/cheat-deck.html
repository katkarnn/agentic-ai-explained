The master decision flow

Start here, every time

Q1 · Can one plain LLM call do it?

→ Just call the model. No agent. Stop.

Q2 · Can you write the steps in advance?

→ WORKFLOW (chaining / routing / parallel).

Q3 · Do the steps depend on what's found at runtime?

→ AGENT (orchestrator / evaluator-loop / subagents).

Q4 · Risky, irreversible, or sensitive?

→ Add guardrails + human-in-the-loop, always.

3 / 27
The one question that ends confusion

Who decides the steps?

🗺️ YOU decide → Workflow

  • Steps fixed ahead of time
  • Predictable, cheap, easy to debug
  • Most "AI products" live here

🧭 The MODEL decides → Agent

  • Model picks its own steps at runtime
  • Flexible, self-correcting
  • Pricier & harder to control

If you can draw the flowchart before running it → workflow. If you can't → agent.

Method · base

Single LLM call

✅ Use when

  • One clear, self-contained ask
  • No tools or live data needed
  • Speed & cost matter most

❌ Don't use when

  • It needs real-world info or actions
  • The task has several dependent steps
  • You need verified, current facts
🧒
EXAMPLE
✅ "Rewrite this sentence to be polite."  ❌ "Book me a flight" (needs tools).

Always the default. Don't add an agent until a plain call clearly can't do it.

Method · workflow

Prompt chaining

✅ Use when

  • Task splits into clear, ordered steps
  • Each step feeds the next
  • You want reliability over flexibility

❌ Don't use when

  • The order changes per request
  • Steps are independent (use parallel)
  • You don't know the steps yet
🧒
EXAMPLE
✅ Draft → make punchier → translate.  ❌ "Handle any customer email" (type varies → routing).
7 / 27
Method · workflow

Routing

✅ Use when

  • Input falls into clear categories
  • Each category needs special handling
  • You can list the categories upfront

❌ Don't use when

  • There's really only one path
  • Categories blur / overlap constantly
  • The right path is unknown until mid-task
🧒
EXAMPLE
✅ Support tickets → billing / tech / refund.  ❌ A free-form research goal with no fixed buckets.
8 / 27
Method · workflow

Parallelization

✅ Use when

  • Subtasks are independent
  • You want speed (run all at once)
  • You want multiple opinions to cross-check

❌ Don't use when

  • Step B needs step A's result (chain instead)
  • Calls share a limited budget/rate limit
  • Merging the results is harder than the task
🧒
EXAMPLE
✅ Check grammar + facts + tone at once.  ❌ "Translate the summary" — needs the summary first.

Method · agent

Orchestrator–workers

✅ Use when

  • You can't know the subtasks in advance
  • A big goal must be split on the fly
  • Subtasks need different skills

❌ Don't use when

  • You already know the steps (use a workflow)
  • The task is small & cheap
  • You can't afford unpredictable cost
🧒
EXAMPLE
✅ "Build a landing page" → lead invents copy/design/image tasks.  ❌ "Resize this image" (one fixed step).
10 / 27
Method · agent

Evaluator–optimizer

✅ Use when

  • Quality really matters
  • You can clearly describe "good"
  • A first draft rarely nails it

❌ Don't use when

  • "Good" is fuzzy / can't be judged
  • Speed & cost beat perfection
  • One pass is already good enough
🧒
EXAMPLE
✅ Polish an essay until score ≥ 0.9.  ❌ "Say hi to the user" (no need to grade-and-retry).

Always cap the retries — or it can loop forever.

Method · agent new

Subagents

✅ Use when

  • A subtask needs lots of reading/searching
  • You want to keep the main context clean
  • Helpers can work independently in parallel

❌ Don't use when

  • The task is small (overkill + extra cost)
  • Subtasks depend tightly on each other
  • You need one continuous conversation
🧒
EXAMPLE
✅ "Audit 200 files" → a helper per batch returns just findings.  ❌ "Answer this one quick question."
12 / 27
Capability · hands

Tool use

✅ Use when

  • The model needs real-world data or actions
  • A reliable function exists for the job
  • Math/lookups it shouldn't guess

❌ Don't use when

  • The model already knows the answer well
  • You'd add 20 tools "just in case" (confuses it)
  • A write-tool is risky with no guardrail
🧒
EXAMPLE
✅ get_weather(), send_email().  ❌ Giving a calculator tool to answer "what's 2+2".

Keep it to 1–5 well-described tools. Fewer, clearer tools = smarter agent.

13 / 27
Capability · hands new

MCP (tool standard)

✅ Use when

  • You want reusable tools across apps
  • Connecting standard services (Drive, GitHub, DB)
  • Building for a team / ecosystem

❌ Don't use when

  • A tiny one-off script (a plain function is fine)
  • An MCP server adds setup you don't need
  • The tool is private & used in one place only
🧒
EXAMPLE
✅ One "GitHub" plug every assistant can reuse.  ❌ A 5-line helper used by a single script.
14 / 27
Capability · memory

RAG

✅ Use when

  • Answers need your private/company docs
  • Facts change & must stay current
  • You want to cut hallucinations

❌ Don't use when

  • The answer is general knowledge
  • The whole doc already fits in context
  • Data changes every second (use a live tool)
🧒
EXAMPLE
✅ "Answer from our HR handbook."  ❌ "What's the capital of France?" (it just knows).
15 / 27
Capability · hands new

Computer / browser use

✅ Use when

  • There's NO API — only a screen/website
  • Legacy apps you can't change
  • Repetitive click-through tasks

❌ Don't use when

  • A clean API exists (use it — faster, safer)
  • Speed & reliability are critical
  • The action is high-risk with no human check
🧒
EXAMPLE
✅ Order on an old supplier site with no API.  ❌ Checking weather (there's a perfect API).

Slowest + most error-prone tool. Last resort, always with a human gate for risky steps.

Technique · thinking new

Planning & reflection

✅ Use when

  • Long, multi-step tasks
  • Mistakes are likely & costly
  • The agent tends to wander off

❌ Don't use when

  • Short, simple tasks (adds cost + delay)
  • Latency matters more than perfection
  • One pass is reliably correct
🧒
EXAMPLE
✅ Plan an essay, then re-check it.  ❌ Re-planning before answering "what's 5×5".
17 / 27
Safety · always consider

Human-in-the-loop

✅ Use when

  • Action costs money / is irreversible
  • Touches sensitive or private data
  • The agent is unsure / low confidence

❌ Skip it when

  • Action is trivially reversible
  • Read-only, low-stakes lookups
  • Asking every time would annoy users
🧒
EXAMPLE
✅ "Confirm before transferring $5,000."  ❌ "Confirm before reading a public webpage."

Rule of thumb: easy to undo → let it run. Hard to undo → ask first.

Choosing memory

Which memory? Pick by question

If you need…UseDon't reach for it when
"What did we just say?"Short-term (context)You need it to survive the run → won't
"What do I know across runs?"Long-term (RAG / agent memory)It's a one-off chat → overkill
"Where did I pause?"Workflow state (checkpoints)The task is instant → not needed
🧒
EXAMPLE
Sticky note = now. Filing cabinet = forever. Bookmark = where you stopped.

Build decision new

Prompt vs RAG vs Fine-tune

Use…WhenAvoid when
PromptingAlmost always — try firstIt needs facts it doesn't have
RAGNeeds your own/fresh factsIt's general knowledge
Fine-tuningFixed style/format or niche skill, at scaleYou haven't maxed out prompt + RAG yet
🧒
EXAMPLE
Ask the friend → give them the book → send them to school. Try the cheap one first!

Order: prompt → RAG → fine-tune. Most problems stop at the first two.

Cost · technique new

Caching

✅ Use when

  • A big fixed prompt repeats every call
  • Loops re-send the same instructions
  • You want lower cost + faster replies

❌ Won't help when

  • Every prompt is totally different
  • The repeated part is tiny
  • Calls are rare & far apart
🧒
EXAMPLE
✅ Same rules before every quiz → write them on the board once.  ❌ A one-time question.
21 / 27
Run-style · technique new

Async / durable / scheduled

✅ Use when

  • The task takes minutes–hours
  • It must survive crashes (durable)
  • It should run on a schedule

❌ Don't use when

  • It's a quick, interactive chat
  • The user is waiting in real time
  • Extra infra isn't worth it
🧒
EXAMPLE
✅ "Each morning, summarize overnight emails."  ❌ "What's 12 × 8?" (answer now).
22 / 27
Safety · non-negotiable new

Guardrails & injection defense

✅ Always when

  • Agent reads web/email/user files
  • It can send, pay, delete, or share data
  • It handles personal/sensitive info

❌ Never skip because…

  • "It worked in testing" (attackers adapt)
  • "It's internal only" (insiders + mistakes)
  • "The model seems smart" (it can be tricked)
🧒
DEFENSES
Least-privilege tools · sandbox actions · treat outside text as data, never commands · guardrails on input AND output.
23 / 27
Quality · non-negotiable new

Evals

✅ Build them when

  • The agent ships to real users
  • You'll keep changing prompts/models
  • Wrong answers have real consequences

❌ Insufficient when

  • You only "eyeball" a few outputs
  • You rely on the model grading itself alone
  • No regression check after changes
🧒
EXAMPLE
Keep an answer key + practice questions. Re-test every time you change your method.

Cost · always

Cost controls (always on)

ControlWhy
Cap max loop turnsStops a runaway agent burning tokens forever
Trim / compact contextDon't resend the whole history each call
Right-size the modelSmall for easy steps, big only for hard ones
Cache fixed promptsPay once for the unchanged part

These aren't optional extras — they're the difference between a demo and a product.

The big "DON'T"

When NOT to build an agent at all

🚫

A workflow fits

You can write the steps → don't hand control to the model.

🚫

One call works

Plain LLM call answers it → no loop needed.

🚫

Zero error tolerance

Use deterministic code, not a probabilistic agent.

🚫

Can't observe / cap it

No tracing, no limits → don't ship it yet.

Agents are powerful but pricier and less predictable. Don't reach for one out of habit.
